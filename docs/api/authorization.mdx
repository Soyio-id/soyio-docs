# Autorizacion

Autorizacion define que operaciones puede ejecutar una API key ya autenticada.

En otras palabras:

- autenticacion valida quien llama
- autorizacion valida si esa key puede ejecutar el endpoint pedido

Si necesitas repasar creacion, uso y renovacion de API keys, vuelve a la [guia de autenticacion](./authentication).

## Modelo de autorizacion

Despues de autenticar el request, Soyio evalua acceso en capas:

1. confirma que la API key este activa
2. revisa los permisos requeridos por el endpoint
3. aplica restricciones del recurso, como existencia o pertenencia
4. aplica reglas adicionales, por ejemplo feature flags o reglas de negocio

La operacion solo se ejecuta cuando todas las capas pasan.

## Donde defines el acceso de una API key

Puedes gestionar permisos desde el [dashboard de Soyio](https://dashboard.soyio.id) o via API al crear y actualizar API keys.

Hoy el acceso de una API key se define con el campo `permissions`.

- Crear API key: [`POST /api_keys`](./resources/create-api-key.api.mdx)
- Actualizar API key: [`PATCH /api_keys/{id}`](./resources/update-api-key.api.mdx)

Ejemplo valido:

```json
{
  "api_key": {
    "label": "Integration Worker",
    "permissions": [
      "consent.api.actions.read",
      "consent.api.manage"
    ]
  }
}
```

Ejemplo invalido:

```json
{
  "api_key": {
    "permissions": ["*.api.manage"]
  }
}
```

:::warning
`*.api.manage` es solo una forma de explicar el patron.

No es un permiso literal. Debes enviar strings exactos, por ejemplo `consent.api.manage`, `dsr.api.write` o `api_keys.api.read`.
:::

## Como se estructuran los permisos

Los permisos actuales siguen un patron modular. Algunos ejemplos reales hoy son:

- `consent.api.read`
- `consent.api.write`
- `consent.api.actions.read`
- `consent.api.actions.write`
- `consent.api.commits.read`
- `consent.api.commits.write`
- `dsr.api.read`
- `dsr.api.write`
- `disclosure.api.read`
- `disclosure.api.write`
- `api_keys.api.read`
- `api_keys.api.manage`

Tambien existen permisos de administracion por modulo, como `consent.api.manage`, `dsr.api.manage`, `roles.api.manage` o `webhooks.api.manage`.

En varios endpoints, el permiso `<modulo>.api.manage` tambien habilita operaciones de ese modulo. No es global para toda la API.

## Como saber que acceso necesita un endpoint

En la referencia de API, cada endpoint puede mostrar un bloque `x-permissions` con `api_any_of`.

Piensa `api_any_of` como: "uno de estos permisos basta".

Ejemplo:

```yaml
x-permissions:
  api_any_of:
    - consent.api.actions.read
    - consent.api.manage
```

Lectura rapida:

| Permisos de la API key | Accede al endpoint |
| --- | --- |
| `consent.api.actions.read` | Si |
| `consent.api.manage` | Si |
| `dsr.api.manage` | No |
| ninguno de los anteriores | No |

## Disena acceso por integracion

Usa el principio de minimo privilegio:

1. lista los endpoints que usa tu servicio
2. revisa los permisos del endpoint en la referencia
3. asigna el permiso mas acotado posible
4. usa `<modulo>.api.manage` solo si realmente necesitas amplitud operativa
5. revisa permisos cada vez que agregues endpoints nuevos a la integracion

## Troubleshooting rapido

| Codigo | Que suele significar |
| --- | --- |
| `401 Unauthorized` | token ausente, invalido, revocado o no autenticable |
| `403 Forbidden` | token valido, pero bloqueado por una regla adicional |
| `404 Not Found` | recurso inexistente o endpoint que oculta acceso no autorizado |

Checklist rapido:

1. confirma que usas `Authorization: Bearer <secret_jwt>`
2. confirma que la API key sigue activa
3. revisa `api_any_of` en la referencia del endpoint
4. verifica que tu key tenga uno de esos permisos exactos
5. revisa si el endpoint tiene reglas extra o si el recurso realmente existe

## Buenas practicas operativas

- usa una API key por servicio
- evita compartir una misma key entre sistemas distintos
- revisa permisos cuando cambie el alcance de la integracion
- monitorea uso de keys y revoca las que ya no se usan
